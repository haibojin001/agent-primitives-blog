What is a primitive?

When people build LLM-based multi-agent systems (MAS), they typically design a bespoke agent team for each new task: custom roles, custom prompts, custom communication protocols. The work doesn't transfer. A code-review pipeline built last week doesn't help when this week's task is competitive analysis.

The right analogy is how we write software. When the same logic appears in five places, you don't copy-paste it five times — you extract it into a function. A primitive is that extracted function for multi-agent systems.

Concretely, a primitive is:

  • A self-contained multi-agent unit that implements one recurring computational pattern
  • It communicates internally via KV cache rather than text, which avoids repeated decoding and reduces information degradation across stages
  • It exposes the same external interface as a single LLM agent — input in, output out — so it can be dropped into any larger system

This library defines three primitives: ReviewPrimitive, VotingPrimitive, and PlanExecutePrimitive. An Organizer can select and compose them automatically for a given query.

The core observation

The primitives were not designed top-down. They were discovered by looking across existing MAS papers and noticing that the same computational shapes kept reappearing under different labels.

Framework Domain What it actually does Underlying pattern
Self-Refine (Madaan et al., 2023) General reasoning Model generates output, then critiques and refines it iteratively Produce → Critique → Revise
Reflexion (Shinn et al., 2023) Decision-making Actor acts, Evaluator scores, Self-Reflection produces verbal feedback for the next trial
MetaGPT (Hong et al., 2023) Software engineering Engineer writes code, QA Engineer tests and reports, Engineer revises

Strip away the domain labels and the same shape appears in all three: one role produces a candidate, another critiques it, the first revises. This recurring function is the Review primitive.

Note A primitive is not defined by its surface appearance (the agent names, the prompts, the domain). It is defined by the computational structure: which agents exist, what they produce, and how their outputs feed into each other. The same structure showing up in ≥ 3 different real systems is the signal that it is worth extracting.

The Review primitive

The ReviewPrimitive implements iterative refinement through critique. Given a task, a Solver agent produces an initial response. A Critic agent evaluates it and produces feedback — not by decoding text from the Solver, but by reading the Solver's KV cache directly. The Solver then refines conditioned on both its own KV cache and the Critic's KV cache, also without any text round-trip.

This KV-cache-based communication has two practical benefits over text-based MAS:

  • Efficiency. The Critic reads the Solver's hidden states directly rather than re-encoding a decoded text string. Token usage is roughly 3–4× lower than text-based equivalents.
  • Robustness. Long-context text communication degrades because LLMs can lose track of instructions buried in a context window. KV cache communication preserves the internal representation without this degradation.

)

Requirements A CUDA GPU is strongly recommended. The primitive loads an LLM and operates on its KV cache. With device_map="auto", it will use all available GPUs. CPU inference is supported but slow.

Full implementation

Here is the complete ReviewPrimitive class. Read through it once — the walk-through below explains every part.

python primitives/ReviewPrimitive.py 
import torch
from transformers import AutoModelForCausalLM, AutoTokenizer
from transformers.cache_utils import DynamicCache
from typing import Optional, Tuple, List


def _pkv_to_legacy(past_kv):
    if isinstance(past_kv, DynamicCache):
        if hasattr(past_kv, "to_legacy_cache"):
            return past_kv.to_legacy_cache()
        raise RuntimeError("DynamicCache incompatible with current transformers.")
    if isinstance(past_kv, (tuple, list)) and len(past_kv) > 0:
        return list(past_kv)
    raise RuntimeError(f"Unsupported cache type: {type(past_kv)}")


def _legacy_to_dynamic(legacy_list):
    return DynamicCache.from_legacy_cache(legacy_list)


@torch.no_grad()
def _forward_step(model, cache, embed):
    out = model(
        inputs_embeds=embed,
        past_key_values=cache,
        use_cache=True,
        return_dict=True,
    )
    logits = out.logits[:, -1, :]
    next_id = torch.argmax(logits, dim=-1)
    next_embed = model.get_input_embeddings()(next_id).unsqueeze(1)
    return next_id, next_embed, out.past_key_values


@torch.no_grad()
def _generate_ids(
    model, tokenizer, cache,
    first_id: torch.Tensor,
    max_tokens: int,
) -> Tuple[List[int], DynamicCache]:
    eos = tokenizer.eos_token_id
    device = model.device

    generated = [first_id.item()]
    if eos is not None and generated[-1] == eos:
        return generated, cache

    cur_embed = model.get_input_embeddings()(first_id.to(device)).unsqueeze(1)

    for _ in range(1, max_tokens):
        next_id, next_embed, cache = _forward_step(model, cache, cur_embed)
        tid = next_id.item()
        generated.append(tid)
        if eos is not None and tid == eos:
            break
        cur_embed = next_embed

    return generated, cache


class ReviewPrimitive:
    def __init__(
        self,
        model_name: str = "Qwen/Qwen3-8B",
        dtype: torch.dtype = torch.float16,
        max_solver_tokens: int = 30000,
        max_critic_tokens: int = 10000,
        device: Optional[str] = None,
    ):
        self.device = device or ("cuda" if torch.cuda.is_available() else "cpu")

        self.tokenizer = AutoTokenizer.from_pretrained(
            model_name, trust_remote_code=True
        )
        self.model = AutoModelForCausalLM.from_pretrained(
            model_name,
            torch_dtype=dtype,
            device_map="auto" if self.device == "cuda" else None,
            trust_remote_code=True,
        )
        if self.device != "cuda":
            self.model = self.model.to(self.device)
        self.model.eval()

        self.max_solver_tokens = max_solver_tokens
        self.max_critic_tokens = max_critic_tokens

        self._solver_init_template = [
            {"role": "system", "content": "You are an intelligent programmer. Complete the function provided by the user."}
        ]
        self._solver_refine_template = [
            {"role": "system", "content": "You are an intelligent programmer."},
            {"role": "user", "content": "Refine your previous solution based on the Critic feedback."},
        ]
        self._critic_template = [
            {"role": "system", "content": "You are an intelligent critic. Provide only feedback."}
        ]

    @torch.no_grad()
    def _solver_initial_kv(self, user_query: str) -> DynamicCache:
        messages = self._solver_init_template + [
            {"role": "user", "content": user_query}
        ]
        prompt = self.tokenizer.apply_chat_template(
            messages, tokenize=False, add_generation_prompt=True
        )
        inputs = self.tokenizer(prompt, return_tensors="pt").to(self.device)
        out = self.model(**inputs, use_cache=True, return_dict=True)
        first_id = torch.argmax(out.logits[:, -1, :], dim=-1)
        _, cache = _generate_ids(
            self.model, self.tokenizer, out.past_key_values,
            first_id, self.max_solver_tokens,
        )
        return cache

    @torch.no_grad()
    def _critic_kv(self, solver_cache: DynamicCache) -> DynamicCache:
        prompt = self.tokenizer.apply_chat_template(
            self._critic_template, tokenize=False, add_generation_prompt=True
        )
        inputs = self.tokenizer(prompt, return_tensors="pt").to(self.device)
        out = self.model(**inputs, use_cache=True, return_dict=True)

        # Concatenate solver KV cache with critic's own KV cache
        solver_legacy = _pkv_to_legacy(solver_cache)
        critic_legacy = _pkv_to_legacy(out.past_key_values)
        merged = [
            (torch.cat([k1, k2], dim=2), torch.cat([v1, v2], dim=2))
            for (k1, v1), (k2, v2) in zip(solver_legacy, critic_legacy)
        ]
        merged_cache = _legacy_to_dynamic(merged)
        first_id = torch.argmax(out.logits[:, -1, :], dim=-1)
        _, final_cache = _generate_ids(
            self.model, self.tokenizer, merged_cache,
            first_id, self.max_critic_tokens,
        )
        return final_cache

    @torch.no_grad()
    def _solver_refine_text(
        self,
        solver_cache: DynamicCache,
        critic_cache: DynamicCache,
    ) -> str:
        prompt = self.tokenizer.apply_chat_template(
            self._solver_refine_template,
            tokenize=False, add_generation_prompt=True,
        )
        inputs = self.tokenizer(prompt, return_tensors="pt").to(self.device)
        out = self.model(**inputs, use_cache=True, return_dict=True)

        # Merge all three KV caches: original solver + critic + refiner prompt
        hist = _pkv_to_legacy(solver_cache)
        crit = _pkv_to_legacy(critic_cache)
        ref  = _pkv_to_legacy(out.past_key_values)
        merged = [
            (torch.cat([k0, k1, k2], dim=2), torch.cat([v0, v1, v2], dim=2))
            for (k0, v0), (k1, v1), (k2, v2) in zip(hist, crit, ref)
        ]
        merged_cache = _legacy_to_dynamic(merged)
        first_id = torch.argmax(out.logits[:, -1, :], dim=-1)
        gen_ids, _ = _generate_ids(
            self.model, self.tokenizer, merged_cache,
            first_id, self.max_solver_tokens,
        )
        return self.tokenizer.decode(gen_ids, skip_special_tokens=True)

    def run(self, user_query: str) -> str:
        solver_cache  = self._solver_initial_kv(user_query)
        critic_cache  = self._critic_kv(solver_cache)
        return self._solver_refine_text(solver_cache, critic_cache)

Walk-through

Helper functions

Three module-level helpers handle KV cache bookkeeping and autoregressive generation. They are shared across all primitives in the library.

_pkv_to_legacy and _legacy_to_dynamic

Newer versions of transformers return a DynamicCache object instead of a plain list of (key, value) tuples. These two helpers normalise between the two representations so the rest of the code can always work with a flat list of per-layer tuples when concatenating, and convert back to DynamicCache before passing back into the model.

_forward_step

A single autoregressive decoding step. It takes the model, the current KV cache, and the embedding of the last predicted token; runs one forward pass; and returns the next token id, its embedding, and the updated cache. It operates on embeddings rather than token ids so that callers can inject any representation into the forward pass.

_generate_ids

Runs _forward_step in a loop to generate a sequence of token ids up to max_tokens. It stops early at eos. It returns both the generated ids and the final KV cache, which is what the next stage of the primitive reads.

Why return the KV cache from generation? The KV cache accumulated during generation encodes the full context the model has seen: the prompt plus every token the model generated. Returning it allows the next stage (e.g. the Critic) to read that full context without re-encoding the decoded text from scratch.

The ReviewPrimitive class

Constructor

The constructor loads the model and tokenizer, sets generation budgets, and defines three fixed message templates:

  • _solver_init_template — the Solver's initial system prompt. Currently specialised for code generation; you can customise this for your domain.
  • _critic_template — the Critic's system prompt. It receives no user message; it reads the Solver's response entirely from the KV cache.
  • _solver_refine_template — the Solver's refinement system prompt and a fixed instruction to revise based on the Critic's feedback.
Customising for your domain To adapt the ReviewPrimitive to a non-coding domain, change the system prompts in __init__. The KV cache mechanism and the three-stage pipeline stay exactly the same. You are only changing what roles the agents play, not the structure.

_solver_initial_kv(user_query)

Runs the Solver on the user query. It applies the chat template, runs a full forward pass to get the first logit, then calls _generate_ids to generate the response. The returned object is the KV cache at the end of generation — a compact representation of everything the Solver produced.

_critic_kv(solver_cache)

This is the key communication step. The Critic has its own system prompt, but its user context is not a decoded text string — it is the Solver's KV cache, concatenated onto the Critic's own KV states layer by layer:

pythonthe KV concatenation
merged = [
    (torch.cat([k1, k2], dim=2), torch.cat([v1, v2], dim=2))
    for (k1, v1), (k2, v2) in zip(solver_legacy, critic_legacy)
]

Concatenating along dim=2 (the sequence dimension) means the Critic attends over both the Solver's full generation context and its own prompt context simultaneously, as if they were one contiguous sequence. The Critic then generates its feedback into its own KV cache, which is what _critic_kv returns.

Same model required KV cache concatenation only works when both agents share the same model architecture and weights. The assumption is that the key and value projections are compatible. The ReviewPrimitive uses a single loaded model for all stages, which satisfies this constraint.

_solver_refine_text(solver_cache, critic_cache)

The final stage. The Solver re-reads both its own original KV cache and the Critic's KV cache, along with a short refinement prompt, all merged together:

pythonthree-way KV merge
merged = [
    (torch.cat([k0, k1, k2], dim=2), torch.cat([v0, v1, v2], dim=2))
    for (k0, v0), (k1, v1), (k2, v2) in zip(hist, crit, ref)
]

The Solver generates the final refined output from this merged context. This output is decoded to text and returned as the primitive's result.

run(user_query)

The public interface. It orchestrates the three stages in sequence:

python
def run(self, user_query: str) -> str:
    solver_cache  = self._solver_initial_kv(user_query)   # Stage 1: solve
    critic_cache  = self._critic_kv(solver_cache)          # Stage 2: critique
    return self._solver_refine_text(solver_cache, critic_cache) # Stage 3: refine

Running it

Basic usage

python
from primitives import ReviewPrimitive

primitive = ReviewPrimitive(model_name="Qwen/Qwen3-8B")

result = primitive.run(
    "Write a Python function that checks if a binary tree is balanced."
)
print(result)

Changing the model

Any model compatible with AutoModelForCausalLM and transformers KV cache works. Larger models generally benefit more from the Critic stage.

python
primitive = ReviewPrimitive(
    model_name="deepseek-ai/DeepSeek-R1-Distill-Qwen-32B",
    dtype=torch.bfloat16,
    max_solver_tokens=30000,
    max_critic_tokens=10000,
)

Adjusting generation budgets

max_solver_tokens controls how long the Solver's initial answer and refinement can be. max_critic_tokens controls how long the Critic's feedback can be. For math benchmarks you may want a large max_solver_tokens; for quick quality checks you can reduce both.

Build your own primitive

The ReviewPrimitive is one extracted function. You can define others for different recurring computational patterns. Here is the process.

  1. Collect 3–5 existing MAS designs in your domain. They can be papers, open-source projects, or pipelines you have built. Draw each one out, with its agents and how they communicate.
  2. Strip away domain labels. Replace "Engineer," "Actor," "Brand Manager" with generic placeholders (Agent A, Agent B). Look at what's left: who produces, who evaluates, who synthesises, who selects.
  3. Find the repeated function. If two or three of your designs implement the same shape under different labels, you have a candidate primitive. A pattern recurring in fewer than 3 real systems is probably not general enough yet.
  4. Define the minimum agent structure. For each agent, write down: what is its job, what does it receive, what does it produce, and what must it not do.
  5. Implement the three-method structure. Each primitive should expose run(query) -> str and implement internal communication via KV cache following the same concatenation pattern as ReviewPrimitive.
Skeleton for a new primitive The minimum structure is: a constructor that loads the model and defines system prompt templates; one method per internal stage that builds or merges KV caches; and a run() method that calls the stages in order.

Composing primitives

Individual primitives can be composed into larger systems. The Organizer does this automatically: given a query, it selects which primitives to use and in what order, optionally guided by a Knowledge Pool of previously effective configurations.

python
from primitives import Organizer

org = Organizer(config={
    "model_name": "Qwen/Qwen3-8B",
    "num_executors": 3,
})

# Returns the composed system's final output
result = org.solve("Write a binary search implementation in Python.")
print(result)

# Inspect the plan without running it
plan = org.plan_only("Write a binary search implementation in Python.")
print(plan)

The Organizer's plan_only() returns a JSON plan describing which primitives to use and in which order. The solve() method executes the plan. The three available primitive types are "review", "vote", and "plan_execute".

Design principles

When adding a new primitive to the library, keep these three properties in mind:

PropertyWhat it meansTest
One function The primitive implements one recurring computational unit. Internal complexity (loops, multiple stages) is allowed; multiple distinct purposes is not. Can you find the same structure in ≥ 3 real MAS designs?
Domain-agnostic roles Agent roles describe what kind of agent it is (Solver, Critic, Planner) not what content it handles. Task knowledge lives in the input. Does the primitive work on tasks from three unrelated domains without code changes?
Bounded internal agents Each internal agent has a non-overlapping, constrained role. The Critic must not produce the final answer. The Planner must not execute. Can you write one sentence describing each agent's job and one sentence describing what it must not do?

Checklist

Before submitting a new primitive to the shared library:

  • The same computational structure appears in at least 3 existing MAS designs
  • The primitive implements one function with a single clear purpose
  • All internal communication uses KV cache concatenation, not text decoding
  • Agent roles are defined by function (Solver, Critic, Planner) not by content domain
  • Each agent has an explicit constraint on what it must not do
  • The run() method accepts a string and returns a string
  • Tested on at least 3 tasks from different domains
  • One-sentence description added to __init__.py __all__ docstring
